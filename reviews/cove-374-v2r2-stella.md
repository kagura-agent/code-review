## Summary

I re-reviewed the v2 rework and specifically checked the two claimed fixes plus the new cleanup code. The deleted-attachment access fix is directionally correct: the attachment GET route checks the `attachments` table before reading from disk, so after message deletion cascades the DB row away, the old URL should return 404 even if unlink cleanup is still pending or fails.

However, the new bulk-delete cleanup has a real correctness bug: it can delete attachment files belonging to messages outside the target channel while leaving their DB rows/messages intact. That is not just hardening; it can break unrelated attachments. I would fix that before merge.

## Claimed Fix Verification

### 1. Orphaned files on message delete — **partially fixed; bulk-delete bug needs changes**

Single-message delete now fetches attachments before deleting the message, then calls `cleanupAttachmentFiles(...)` after a successful delete. Channel clear similarly fetches `getByChannelId(channelId)` before deleting all channel messages. Those paths look reasonable as best-effort cleanup.

The bulk-delete path is unsafe:

- It loops over every requested `body.messages` ID and calls `repos.attachments.getByMessageId(msgId)` before knowing whether that message belongs to the current channel.
- The actual deletion correctly uses `repos.messages.delete(channelId, msgId)`, so messages from other channels are not deleted.
- But if at least one message in the request is deleted, cleanup runs for **all** attachments collected earlier, including attachments from non-deleted messages in other channels.

Impact: a malformed or malicious bulk-delete request containing `[validMessageInThisChannel, messageFromOtherChannelWithAttachment]` can unlink the other channel's attachment file while the other message and attachment DB row remain. The attachment URL will still pass the DB existence check but return 404 because the file is gone.

Recommended fix: only collect attachments for messages proven to belong to the target channel and actually selected for deletion. For example, query attachments by `(message_id, channel_id)` before delete, or first filter/validate message IDs in the channel, collect attachments for that filtered set, then delete that same set.

Minor cleanup notes: `cleanupAttachmentFiles()` unlinks the file but leaves empty attachment/channel/guild directories behind; that is acceptable for now. It also swallows errors, which is fine for fire-and-forget cleanup but means a later janitor/reconcile task would still be useful.

### 2. Deleted attachments still accessible — **fixed for deleted rows**

The GET route now checks `repos.attachments.exists(safeAttachmentId)` before `readFile`. Since the attachment table has `FOREIGN KEY (message_id) REFERENCES messages(id) ON DELETE CASCADE`, deleting a message removes the row immediately. That means an old URL should stop serving as soon as the message delete commits, even if file unlink fails asynchronously.

Two nits, non-blocking here:

- The existence check is duplicated in `app.ts`.
- `exists(attachmentId)` only checks the ID, not that the DB row's guild/channel/filename match the URL path. With the current one-file-per-attachment storage layout this is not an immediate blocker, but a `getByAttachmentId()` returning canonical metadata would be cleaner and less error-prone.

## R1 Additional Issues Status

Per the small-team/non-blocking guidance, I am not treating these as merge blockers, but their status is:

- **Upload not atomic with DB write:** still present. Files are written before `repos.messages.create(...)` and `repos.attachments.createMany(...)`; a later DB failure can leave unreferenced files. Follow-up recommended.
- **Snowflake IDs are enumerable:** still present. The URL access model relies on attachment snowflakes rather than signed URLs/random secrets. Acceptable only if this is an intentional small-team tradeoff.
- **MIME trusts `file.type`:** still present. The server allowlist checks client-supplied multipart MIME instead of magic bytes. Follow-up recommended before broader deployment.
- **Sanitize inconsistency:** still present. Upload replaces invalid filename chars with `_`; GET strips invalid chars. Not a blocker, but extracting one shared helper would reduce edge cases.
- **Duplicate `isImage = true`:** still present in the `.webp` branch. Harmless cosmetic nit.

## Fresh Cleanup Code Review

The URL parsing in `cleanupAttachmentFiles()` matches the generated `/api/v10/attachments/{guildId}/{channelId}/{attachmentId}/{filename}` shape and decodes the filename before calling `getAttachmentPath()`. Because the input comes from stored attachment URLs generated by this server, path traversal risk is low in normal operation.

The main path/correctness issue is the bulk-delete collection described above: cleanup must be scoped to attachments of messages actually deleted from the target channel.
